# План рефакторинга фронтенда (T-Mebel / migration)

Цель: один предсказуемый источник правды для данных и локали, предсказуемая структура кода, отсутствие дублей маршрутов и компонентов. План объединяет исходные задачи, переход на **Feature-Sliced Design (FSD)** и сопутствующие инженерные практики.

---

## 0. Архитектура: Feature-Sliced Design (FSD)

### 0.1 Почему FSD

- Явные **слои** и **направление зависимостей** (сверху вниз: `app` → `pages` → `widgets` → `features` → `entities` → `shared`).
- Фичи и сущности не смешиваются с роутингом Next.js.
- Проще удалять мёртвый код и онбордить людей.

### 0.2 Соглашение для этого репозитория (Next.js App Router)

| FSD-слой   | Назначение | Предлагаемое расположение |
|------------|------------|----------------------------|
| **app** (bootstrap) | Провайдеры, глобальные стили, инициализация, то что крутит всё приложение | `src/app/` *как корень Next* + при необходимости `src/application/` или `shared/lib/providers` для композиции |
| **pages**  | Композиция страницы из виджетов (тонкий слой «сценария экрана») | Слайсы вида `src/pages/home`, `src/pages/service-details` — **импортируются только из `src/app/**/page.tsx`** |
| **widgets** | Крупные автономные блоки UI (шапка, футер, галерея товара целиком) | `src/widgets/*` |
| **features** | Законченные пользовательские действия (смена языка, отправка отзыва, CTA) | `src/features/*` |
| **entities** | Бизнес-сущности: типы, мапперы, мелкие UI по сути (карточка отзыва) | `src/entities/product`, `src/entities/review`, … |
| **shared** | Переиспользуемое без предметной области: UI-kit, `api` client, config, утилиты | `src/shared/ui`, `src/shared/api`, `src/shared/config`, `src/shared/lib` |

**Важно:** каталог `src/app/` остаётся **только для роутинга и layout/metadata** Next.js. Логика и разметка страниц живут в `pages` (FSD) / `widgets` / `features`, а не размазываются по десятку `page.tsx`.

### 0.3 Правила импортов (кратко)

- Слайс импортирует только из слоёв **ниже** себя (`widgets` → `features`, `entities`, `shared`, но не наоборот).
- Внутри слоя — публичный API слайса через `index.ts` (re-export), без глубоких импортов `../../../`.
- `shared` не импортирует из `entities` и выше.

### 0.4 Публичные API и именование

- В каждом слайсе: явный `index.ts` (что можно импортировать снаружи).
- Типы сущностей: `entities/<name>/model/types.ts` (или `model/index.ts`).

## 1. Почистить фронт: неиспользуемое и дубли

### 1.1 Инвентаризация

- [ ] Построить граф импортов: найти файлы в `src/components`, `src/app`, которые **нигде не импортируются**.
- [ ] Отдельно отметить **дубли маршрутов** (например `/service/[id]` vs `/[locale]/service/[id]`) и **дубли компонентов** одного сценария (`FullPage` vs `ProductGallery` для карточки товара).

### 1.2 Удаление / консолидация

- [ ] Удалить или слить неиспользуемые компоненты и стили.
- [ ] Оставить **один** пользовательский поток «страница товара»: один роут (с учётом i18n), один виджет деталки.
- [ ] Проверить неиспользуемые зависимости в `package.json` после чистки (опционально `depcheck`).

### 1.3 Инструменты

- [ ] ESLint: `unused-imports` / правила на неиспользуемые переменные и импорты.
- [ ] По желанию: анализ бандла (`@next/bundle-analyzer`) после крупных удалений.

---

## 2. Рефакторинг файлов и компонентов, типы

### 2.1 Глобальные и локальные типы

- [ ] **Глобальные** договоры с API: `Product`, `Review`, `Image`, ответы ошибок — в `entities/*/model` или `shared/api/types` (если действительно cross-cutting).
- [ ] **Локальные** типы пропсов и состояния виджета/фичи — рядом с компонентом или в `model` слайса, **не** в общий `types.ts` без необходимости.
- [ ] Убрать `as any` в метаданных и API; ввести схему валидации ответа (например Zod) на границе `shared/api`.

### 2.2 Структура после миграции

- [ ] Перенести текущие `src/components/*` в соответствующие слои FSD (`widgets`, `features`, `entities`, `shared/ui`).
- [ ] Единый стиль именования папок и файлов (PascalCase для компонентов или kebab-case — зафиксировать один раз).

### 2.3 Стили

- [ ] CSS Modules остаются рядом с компонентом или в `ui/Component/Component.module.css` по соглашению.

---

## 3. API: отдельный слой и кэширование

### 3.1 Единый API-слой

- [ ] Базовый URL и префиксы — из `shared/config` / env (`NEXT_PUBLIC_*`), без хардкода в компонентах.
- [ ] Один модуль `shared/api/client`: `fetch` или axios с перехватчиками, разбор ошибок, опционально передача `locale` (query/header) для бэкенда.
- [ ] Методы по доменам: `entities/product/api/getProduct.ts` или `shared/api/products.ts` — зафиксировать и не плодить дубли `fetch` в `useEffect`.

### 3.2 Кэширование

- [ ] **Сервер (Next):** для данных из API решить политику: `fetch` с `next: { revalidate }` / SSR только где нужно; не дублировать смысл кэша клиента и CDN вслепую.
- [ ] **Клиент:** TanStack Query (React Query) или аналог — staleTime, ключи с `locale` и `id`, отмена запросов при смене страницы.
- [ ] Договориться с бэкендом про заголовки (`Cache-Control`, `Vary`) если ответ зависит от языка.

### 3.3 Контракт с бэкендом

- [ ] По возможности: OpenAPI / общая схема; минимум — Zod на фронте для ответов `GET /product/:id`.
- [ ] Долгосрочно: тексты товара с бэка по `locale` (см. раздел 6).

---

## 4. Клиентские и серверные компоненты

### 4.1 Принципы

- [ ] По умолчанию — **Server Components**; `'use client'` только где нужны хуки, браузерные API, интерактив.
- [ ] Данные для SEO и первого рендера — по возможности получать на сервере и передавать вниз пропсами.

### 4.2 Конкретика для текущего проекта

- [ ] Страницы в `app/**/page.tsx` — тонкие: `generateMetadata`, вызов API/данных, сборка из слоёв FSD.
- [ ] Карусели, модалки, формы — клиентские виджеты/фичи с минимальной областью `'use client'`.
- [ ] Убрать дублирование `NextIntlClientProvider` с жёстко прошитым `uk` в корневом layout там, где ломается согласованность с `[locale]` (привести к одной модели).

## 5. Дополнительно (интегрировать в тот же проход)

### 5.1 Роутинг и i18n

- [ ] Одна матрица URL: документировать `localePrefix: 'as-needed'` и редиректы в `middleware`.
- [ ] `generateMetadata` и тело страницы — **один источник** текстов для SEO и UI по данному маршруту.

### 5.2 Обработка ошибок и UX

- [ ] `error.tsx` / `loading.tsx` на ключевых сегментах при необходимости.
- [ ] Тосты и сообщения об ошибках — через i18n-ключи, без захардкоженного русского в одном месте и английского в другом.

### 5.3 Изображения

- [ ] `next/image`: `remotePatterns` для домена с медиа; единая политика `alt` (лежит в переводах или приходит с бэка).

### 5.4 Формы

- [ ] Единый стек (уже есть RHF): добавить схемы Zod, переиспользуемые поля там где дубли.

### 5.5 Безопасность

- [ ] Не светить секреты в клиентском бундле; вызовы админки — через server-only пути при необходимости.

### 5.6 Линтинг и граф зависимостей

- [ ] Запрет циклических импортов (`eslint-plugin-import` + правило, либо `madge`).
- [ ] Соглашение по порядку импортов (опционально).

---

## 6. Синхронизация с бэкендом (после или параллельно фронту)

- [ ] Продуктовые `title` / `description` / SEO-поля — с API по выбранной локали (без дублирования тысяч ключей в `messages/*.json` для каждого id).
- [ ] В `messages/*.json` оставить **оболочку UI** (кнопки, лейблы, общие фразы).

---

## 7. Рекомендуемый порядок выполнения

1. **Зафиксировать целевую модель URL и i18n** + убрать дубли маршрутов/страниц товара (иначе очистка компонентов будет бить не в то).
2. **Ввести FSD-скелет** (`shared`, `entities`, затем перенос по одному виджету).
3. **API-слой + типы + env**; перевести вызовы с компонентов в один client.
4. **Разделение server/client** для ключевых страниц.
5. **Кэш** (серверные политики + клиентский React Query при необходимости).
6. **Чистка неиспользуемого** и миграция контента товаров на бэк по мере готовности API.

---

## 8. Критерии готовности (definition of done)

- Один сценарий «карточка товара» без двух разных компонентов-реализаций за разными URL.
- Нет «висячих» файлов без импортов в прод-сборке (по согласованному чек-листу).
- Все внешние URL и секреты вынесены в config/env.
- Импорты соблюдают слои FSD; публичные API слайсов задокументированы через `index.ts`.
- Локаль на странице согласована с `next-intl` и с запросами к API.
